## 2. 사고의 모양 → 포맷 (강제 아님, **메뉴**)

> [!warning] 자동 다이어그램화 금지. 대부분의 답은 3줄 산문·리스트가 정답이다.
> 아래는 "이 모양이면 이 포맷이 어울린다"는 **선택지**지, 매 출력 강제가 아니다.
> 형식이 사고를 대신하기 시작하면(골격 그대로 두고 간격만 만지는 루프) 멈춘다.

| 사고의 모양 | 어울리는 포맷 |
|---|---|
| 분해(하위로 쪼개기) | Mermaid `mindmap` |
| 흐름·인과·시간·분기 | Mermaid `flowchart` / `sequenceDiagram` (분기는 다이아몬드) |
| 연상·연결망 | 원자 노트 + `[[위키링크]]` + 그래프 뷰 |
| 공간 배치 | Canvas `.canvas` — **필요할 때만** (수기 JSON은 비용이 크다) |

- 마인드맵은 트리라 계층만 보인다. 옆으로 이어지는 연결(예: `[[근태 L4]]` ↔ `[[PayTwin 법령엔진]]`)은 **위키링크 + 그래프 뷰**로 드러낸다.
- "사고 + 연결망을 매번 둘 다" 뽑지 않는다. 그 과제에 필요한 쪽만.

## 3. 노트 위생

- **원자 노트**: 개념·결정·리스크 하나당 노트 하나. 길어지면 쪼개고 링크로 잇는다.
- **링크 우선**: 새 개념 등장 즉시 본문에서 `[[개념]]`으로 연결. **고아 노트 금지.**
- **MOC(Map of Content)**: 큰 주제는 허브 노트 하나가 하위를 `[[ ]]`로 모은다.
- **프론트매터**: 모든 노트 상단에

  ```
  ---
  tags: [기획, 근태, SaaS]
  status: draft | active | archived
  ---
  ```

## 4. 서술 규율

- **결론을 맨 위로** (lead with the answer). 빌드업 후 결론 금지.
- **트레이드오프는 표로.** 산문으로 얼버무리지 않는다.
- **비판이 가치.** 칭찬·메아리 금지. 약점을 먼저 말한다.
- **콜아웃 컨벤션** — 사고의 질감을 색으로:

  | 용도 | 콜아웃 |
  |---|---|
  | 핵심 통찰 | `> [!tip]` |
  | 의사결정 | `> [!note]` |
  | 리스크·코너케이스 | `> [!warning]` |
  | 막힌 지점·미해결 | `> [!question]` |
  | 반례·반박 | `> [!danger]` |

## 5. 설명·분해 순서 (전역과 통일)

- 개념 설명은 **비유 → 벤치마킹 사례 → 실전응용** 3단 (전역 CLAUDE.md와 동일).
- **"보통 회사 vs 컨설팅 회사"** 대비는 필요할 때 끼우는 **비교 렌즈**지, 별도 4번째 단계가 아니다.
- 기획·제품 분해는 **BMAD(분석 → 기획/PRD → 설계 → Epic/Story) → 4계층(L1 데이터기반 → L2 진실원천[질문→지표] → L3 스킬 → L4 검증)** 순서.
- 외피(고객 대상) 문서는 **목적 → 내용 → 첨부** 양식. 캐주얼 대화나 "양식" 호출 시 제외.
