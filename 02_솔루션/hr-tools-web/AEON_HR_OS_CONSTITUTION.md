# AEON HR OS 개발 헌법

> 정보는 기반이고, Synapse는 신경망이며, AI는 판단지능이다.

AEON HR OS는 단순한 HR 관리 시스템이 아니다.  
조직 안에서 발생하는 사람 관련 사건을 감지하고, 연결하고, 해석하고, 기록하고, 학습하여 더 나은 인사 의사결정을 가능하게 하는 **HR 운영지능**이다.

---

## 1. Product Vision

AEON HR OS의 목표는 단순히 채용, 근태, 평가, 보상, 퇴직 기능을 각각 만드는 것이 아니다.

본 시스템은 다음 흐름을 중심으로 설계된다.

```
Information
→ Event
→ Synapse
→ Workflow
→ Decision
→ Record
→ Learning
→ Prediction
→ Autonomous HR Operation
```

즉, AEON HR OS는 HR 정보를 정리하는 시스템을 넘어,  
HR 사건을 연결하고, 업무를 반응시키며, 판단과 기록을 추적해 AI가 학습 가능한 구조를 만드는 것을 목표로 한다.

---

## 2. Core Philosophy

모든 HR 업무는 사건으로 시작된다.

사건은 연결을 만들고,  
연결은 업무를 반응시키며,  
업무는 판단을 요구하고,  
판단은 기록으로 남으며,  
기록은 다음 AI의 학습 재료가 된다.

따라서 모든 기능은 단순 CRUD가 아니라 다음 구조 안에서 설계되어야 한다.

```
Event → Workflow → Decision → Record → Learning
```

기능을 만들 때는 반드시 다음 질문에 답해야 한다.

1. 이 기능은 어떤 HR 정보를 다루는가?
2. 이 기능에서 어떤 HR 사건이 발생하는가?
3. 해당 사건은 어떤 모듈로 전달되어야 하는가?
4. 어떤 후속 업무가 생성되거나 갱신되어야 하는가?
5. 사람이 어떤 판단을 해야 하는가?
6. 그 판단의 근거는 어떻게 기록되는가?
7. 이 기록이 향후 AI 학습 재료가 될 수 있는가?

## 3. Non-Negotiable Principles

### 3.1 정보 구조가 먼저다

AI 기능보다 데이터 모델과 사건 정의를 우선한다.  
불분명한 정보 구조 위에 AI를 올리지 않는다.

```
잘못된 정보 구조 위에 AI는 증폭기가 아니라 오류 증폭기다.
```

---

### 3.2 모든 주요 상태 변경은 Event로 기록한다

다음과 같은 상태 변경은 단순 데이터 수정이 아니라 HR Event로 본다.

- 채용의뢰 승인
- 채용공고 생성
- 지원자 선정
- 서류합격
- 면접 완료
- 최종합격
- 처우협의 완료
- 입사일 확정
- 입사 완료
- 직무 변경
- 조직 변경
- 발령
- 퇴직 신청
- 복직
- 평가 완료
- 보상 변경
- 퇴직 예정
- 퇴직 완료

---

### 3.3 모듈 간 직접 결합을 피하고 Synapse Bus를 통해 연결한다

채용, 온보딩, 인사마스터, 근태, 평가, 보상, 퇴직 모듈은 서로 직접 강하게 묶지 않는다.  
각 모듈은 Event를 발행하고, 필요한 모듈은 Event를 구독한다.

```
Module → Event → HR Synapse → Subscriber Modules
```

---

### 3.4 AI는 독립 기능이 아니라 Synapse 위에서 작동하는 판단 보조 계층이다

AI는 맥락 없는 문장 생성기가 아니다.  
AI는 HR 사건, 직무 정보, 사람 정보, 조직 정보, 제도 정보, 과거 판단 기록을 바탕으로 제안·검토·예측을 수행해야 한다.

---

### 3.5 사람의 최종 판단권을 보존한다

시스템이 추천하고, 사람이 승인한다.  
중요한 인사 판단에는 반드시 근거와 로그를 남긴다.

대상 예시:
- 경력 우선
- 처우협의
- 최종합격
- 평가등급 조정
- 보상 조정
- 징계 검토
- 퇴직 처리
- 근태 예외 인정

---

### 3.6 모든 판단은 학습 가능한 형태로 남긴다

판단 결과만 저장하지 않는다.  
판단 근거, 참고 데이터, 예외 사유, 승인자, 승인 시각, 관련 문서를 함께 기록한다.

## 4. Information & Requirement First

AEON HR OS의 가장 중요한 기반은 정보와 사건이다.

정보와 사건이 견고하지 않으면 Synapse는 부실해지고,  
Synapse가 부실하면 AI는 잘못된 맥락을 학습하게 된다.

따라서 모든 개발은 다음 순서를 따른다.

1. 정보 정의
2. 사건 정의
3. 이벤트 정의
4. 업무 흐름 정의
5. 판단 기준 정의
6. 기록 구조 정의
7. AI 개입 지점 정의

AI 기능은 언제나 이 기반 위에 올라가야 한다.

## 5. HR Synapse

HR Synapse는 AEON HR OS의 이벤트 버스이자 업무 연결 신경망이다.

HR Synapse의 역할은 다음과 같다.

- HR 데이터 변경을 의미 있는 사건으로 감지한다.
- 사건을 관련 모듈에 전달한다.
- 후속 업무를 자동으로 생성하거나 제안한다.
- AI가 사건의 맥락을 해석할 수 있도록 구조화된 정보를 제공한다.
- 모든 사건과 판단 이력을 HR Memory에 기록한다.

## 6. Core Event Examples

### 6.1 RecruitmentRequestApproved

채용의뢰가 승인되었을 때 발생한다.

필수 동작:
- 채용공고 이후 작업 생성
- 직무기술서 기준 공고문 자동 구성
- 서류평가 기준 작성
- 면접 질문 후보 작성
- 채용 일정 태스크 생성
- 담당부서 알림

---

### 6.2 ApplicantHired

지원자가 최종합격 처리되었을 때 발생한다.

필수 동작:
- 온보딩 태스크 생성
- 근로계약서 이후 작성
- 입사안내 발송
- 사번 생성 준비
- 근태 대상자 등록 예약
- 담당부서 알림
- 입사 30/60/90일 체크 태스크 생성

---

### 6.3 JobDescriptionUpdated

직무기술서가 수정되었을 때 발생한다.

필수 동작:
- 채용공고 주요업무 갱신 제안
- 서류평가 기준 갱신 제안
- 면접질문 갱신 제안
- 온보딩 과정 갱신 제안
- 평가 KPI 후보 갱신 제안
- 관련 구성원 역량 갭 분석

---

### 6.4 EmployeeJoined

입사자가 실제 입사 처리되었을 때 발생한다.

필수 동작:
- 인사마스터 활성화
- 근태 대상자 등록
- 급여 대상자 등록 준비
- 평가 대상자 생성
- 조직도 반영
- 권한 부여 요청
- 온보딩 진행률 추적 시작

---

### 6.5 EmployeeResigned

퇴직 신청 또는 퇴직 완료 시 발생한다.

필수 동작:
- 퇴직금 계산 태스크 생성
- 인수인계 태스크 생성
- 계정 및 권한 회수 요청
- 퇴직면담 기록 생성
- 조직 결원 리스크 분석
- 후임 채용 필요 여부 검토

---

## 7. AI Layer

AEON HR OS에서 AI는 Synapse 위에서 작동한다.

AI의 역할은 다음과 같다.

| 역할 | 설명 |
|---|---|
| 작성 보조 | 공고문, 메일, 보고서, 계약서 이후 작성 |
| 분류 보조 | 이력서, 직무, 리스크 유형 분류 |
| 판단 보조 | 경력우선, 평가 코멘트, 근태 예외 검토 |
| 추천 보조 | 면접질문, 온보딩 과정, 교육과정 추천 |
| 감지 보조 | 퇴사위험, 조직공백, 제도 리스크 감지 |
| 실행 보조 | 태스크 생성, 알림, 승인흐름 제안 |

AI는 반드시 다음 정보를 참조해야 한다.

- 조직 정보
- 직무 정보
- 사람 정보
- 채용 정보
- 평가 정보
- 근태 정보
- 제도 정보
- 과거 판단 기록
- 관련 Event Log

## 8. HR Memory

HR Memory는 사건, 판단, 근거, 결과를 추적하는 저장소다.

저장 대상:
- Event Log
- Workflow Log
- Decision Log
- Approval Log
- Exception Reason
- AI Suggestion
- Human Feedback
- Final Outcome

HR Memory는 향후 AI 학습과 HR Digital Twin의 기반이 된다.

## 9. Long-Term Blueprint

AEON HR OS의 장기 목표는 다음 단계로 진행하는 것이다.

### 9.1 HR Data Layer
조직, 직무, 사람, 제도, 업무 데이터를 구조화한다.

### 9.2 HR Synapse
HR 사건을 감지하고 모듈 간 업무를 연결한다.

### 9.3 HR Cortex
AI가 사건의 맥락을 이해하고 판단을 보조한다.

### 9.4 HR Agent
채용, 온보딩, 근태, 평가, 보상, 퇴직 영역별 전문 Agent가 작동한다.

### 9.5 HR Digital Twin
조직 변화와 인력 이동을 시뮬레이션한다.

### 9.6 Autonomous HR OS
시스템이 인사 리스크와 후속 업무를 예측하고 제안한다.

### 9.7 Organization Intelligence
조직의 목적, 직무, 사람, 제도, 성과를 종합적으로 이해하는 HR 운영지능으로 진화한다.

| 영역 | 설명 |

## 11. Development Instructions for AI Agents

기능을 구현할 때 고립된 화면으로 취급하지 않는다.

항상 다음을 식별한다:

1. 이 기능은 어떤 HR 정보를 관리하는가?
2. 여기서 어떤 HR 사건이 발생할 수 있는가?
3. 해당 사건에 어떤 모듈이 반응해야 하는가?
4. 어떤 워크플로우가 생성 또는 갱신되어야 하는가?
5. 사람이 어떤 결정을 내려야 하는가?
6. 어떤 근거나 이유가 기록되어야 하는가?
7. 이 기록이 미래 AI 학습 재료가 될 수 있는가?

정보, 이벤트, 워크플로우, 결정, 기록에 연결되지 않는 기능은 HR OS 핵심에 속하는지 재검토한다.

## 12. Architecture Rules

### 12.1 Database

- 모든 주요 Entity는 생성자, 수정자, 생성일, 수정일을 가진다.
- 모든 주요 상태 변경은 Event Log로 남긴다.
- 판단이 필요한 업무는 Decision Log를 가진다.
- 예외 처리가 필요한 업무는 Exception Reason을 가진다.
- AI 제안은 최종 판단과 분리하여 저장한다.

---

### 12.2 API

API는 단순 CRUD만 제공하지 않는다.  
상태 변경 API는 반드시 Event 발행 여부를 검토한다.

예시:
```
POST /api/recruitment-requests/{id}/approve
→ RecruitmentRequestApproved Event 발행

POST /api/applicants/{id}/final-hire
→ ApplicantHired Event 발행
```

---

### 12.3 Frontend

화면은 단순 입력창이 아니라 업무 흐름을 보여줘야 한다.

각 화면은 가능한 한 다음 정보를 포함한다:
- 현재 상태
- 다음 단계
- 관련 Event
- 필요 태스크
- AI 제안
- 담당자 판단 필요 여부
- 기록된 근거

---

### 12.4 AI

AI는 단독으로 최종 결정을 내리지 않는다.  
AI는 제안, 요약, 검토, 위험 감지, 추천을 수행한다.  
최종 판단은 사람이 수행하고, 그 판단이 기록된다.

향후 AI와 Agent가 학습할 수 있는 **HR 사건 로그와 판단 구조**다.

## 15. Final Operating Principle

AEON HR OS의 모든 개발은 아래 원칙에 따른다.

```
정보를 정확히 정의한다.
상태 변경은 사건으로 본다.
사건은 Synapse로 연결한다.
후속 업무를 Workflow로 만든다.
중요 판단은 사람이 승인한다.
판단 근거는 기록한다.
기록은 AI의 학습 재료가 된다.
AI는 다음 더 나은 판단을 제안한다.
```

이 구조를 벗어나는 기능은 AEON HR OS의 핵심 철학에 맞지 않는다.
