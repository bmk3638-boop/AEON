## 0. 용어정리 & 대조표 (비개발자용)

| 전문용어 | 쉬운 뜻 | 비유 |
|---|---|---|
| **백엔드(FastAPI)** | 화면 뒤에서 계산·저장하는 두뇌 | 건물의 기계실 |
| **프론트엔드(React)** | 사람이 보는 화면 | 건물의 사무실·로비 |
| **DB(Postgres/Supabase)** | 데이터 창고 | 서류 보관실 |
| **API / 라우터** | 화면과 두뇌가 대화하는 창구 | 민원 창구 |
| **모델/테이블** | 데이터를 담는 서랍 양식 | 서류 양식지 |
| **마이그레이션** | 서랍 구조를 바꾼 기록 | 창고 리모델링 이력 |
| **공통코드** | 모든 부서가 쓰는 표준 숫자 이름표 | 표준 품번 |
| **HR Synapse(시냅스)** | 한 곳 입력→관련 부서 자동 전달 | 사내 자동 배송관 |
| **Agent Action(승인 큐)** | AI가 제안, 사람이 도장 찍어야 실행 | 결재 대기함 |
| **SynapseRun(원장)** | 뭐가 언제 어디로 갔는지 추적 기록 | 택배 배송 추적 |
| **AuditLog(감사)** | 지울 수 없는 영구 기록 | 블랙박스 |
| **LLM(Gemini)** | 글을 써주는 AI | 똑똑한 비서 |
| **게이트웨이** | AI를 한 곳에서만 연결(갈아끼우기 쉬움) | 멀티탭 콘센트 |
| **Context Radar** | 문서·코드에서 근거를 찾아주는 검색 | 사내 위키 검색 |
| **RAG** | AI가 근거를 찾아 붙여 답하는 방식 | 자료 찾아보고 답변 |
| **온톨로지/지식그래프** | 개념·관계를 잇는 지도 | 조직도+관계도 |
| **GraphRAG** | 그 관계지도로 근거를 찾는 것 | 관계도 보고 답변 |
| **QA Gate** | 출시 전 검사대 | 출고 전 품질검사 |
| **PII 마스킹** | 개인정보 가리기 | 서류 검은 칠 |
| **3모드** | 설계자/관리자/직원 권한 분리 | 본사/지점/고객 출입증 |
| **멀티테넌트** | 고객사별 데이터 격리 | 한 건물, 층마다 다른 회사 |
| **배포(Deploy)** | 인터넷에 올려 접속 가능하게 | 가게 개업 |
| **CI/빌드** | 코드를 실제 돌아가는 앱으로 굽기 | 반죽→빵 |
| **BMAD** | 기획→설계→작업지시 방법론 | 건축 설계 프로세스 |

### 상태 등급 대조 (대표님 판단 기준을 표준화)
| 등급 | 뜻 |
|---|---|
| **Active** | 실제 API+데이터+화면이 다 있고 작동 |
| **Active/확장 중** | 코어는 되는데 세부 기능 계속 붙이는 중 |
| **Partial** | 화면·백엔드가 섞여 있고 일부만 진짜 작동 |
| **Candidate** | 연구·미래 확장(아직 본격 구현 아님) |
| **[설계]** | 문서·계획만 있고 코드는 없음 |

## 1. 한 장 재보고 (핵심만)

- **정체**: "채용 자동화 웹앱"을 넘어선 **HR 운영 OS의 MVP**. HR 전 과정(확보→재직→성장→보상→관계→이직)을 한 시스템으로.
- **규모**: 15개 업무 파트, 화면 ~60장, 데이터 서랍 49개, 백엔드 창구 35개.
- **완성도는 파트마다 다름**:
  - **가장 강함(Active)**: 확보관리(채용·사람인·AI지원자분석·입사전환), 조직/발령, 근태, 컴플라이언스, 공통코드, 감사, HR Synapse.
  - **확장 중**: 노무, 보상, 유지.
  - **Partial**: 평가, 교육, 이직 일부(화면·백엔드 섞임).
  - **[설계]**: 지식그래프·GraphRAG·온톨로지·Agent Harness(문서·8주 플랜).
- **차별점 한 문장**: **"화면 묶음이 아니라, HR 이벤트가 다른 업무로 자동 전파되고, AI는 제안·사람은 승인·모든 건 기록으로 남는 운영 OS."**
- **AI**: 제품은 **Gemini 한 개**(갈아끼우기 쉬운 구조). MCP·플러그인·BMAD는 **개발 도구**이지 제품 기능 아님(면접 주의).

## 2. 앞으로의 방향 · 필요 기능 · 부족한 기능 보강 (가장 중요)

### 2-1. 전략 방향 (한 문장)
> **"넓게 벌린 15개 파트 중, 돈 되고 증명된 3~4개를 '진짜 완성'시켜 1차 출시하고, 나머지는 로드맵으로 남긴다."** — 넓힘(breadth)에서 **깊이(depth)**로 전환.

### 2-2. 지금 부족한 것 (Gap) — 우선순위순
| # | 부족한 것 | 왜 문제 | 보강 방향 |
|---|---|---|---|
| G1 | **완성도 편차** (평가·교육·이직 Partial) | "다 된다"고 보여주면 데모 중 깨짐 | 출시 범위를 Active 파트로 **좁힘**(스코프 확정) |
| G2 | **급여 실제 금액 로직** (밴드 금액 미구축) | 보상 데모가 "지급대기"까지만 | 2026 최저임금 확정치 + 직군 밴드표 구축 |
| G3 | **멀티테넌트 격리** (데모DB 공유, 테스트DB 미분리) | 고객사별 데이터 섞임 위험 | 테넌트별 DB/스키마 분리(출시 전 필수) |
| G4 | **배포·운영** (지금 로컬만) | 웹에서 못 봄 → 아래 4번 | 클라우드 배포(프론트/백/DB) |
| G5 | **보안 하드닝** (침투테스트·비밀관리 미완) | SaaS 신뢰성 | env 비밀관리, 권한 회귀, 감사 |
| G6 | **LLM 운영 정책** (키·비용·민감정보) | 개인정보 외부전송 리스크 | 민감정보 마스킹 후 전송, 비용 상한 |
| G7 | **전 OS 데이터 내보내기** (일부만) | "데이터는 고객 자산" 원칙 | 표준 Excel export + 이관 가이드 |
| G8 | **E2E 자동 테스트/증거** | 출품 신뢰성 | Playwright 시나리오 녹화 |
| G9 | **지식그래프/GraphRAG** [설계] | 차별점의 미래 | 1차=발령그래프부터 착공(출시 후) |
| G10 | **사람인 자동화 배포** (브라우저 필요) | 서버리스에 못 올림 | 별도 워커/데스크톱 컴포넌트로 분리 |

### 2-3. 새로 필요한 기능 (제조업/경동나비엔 대비)
- **교대·근무표 자동화**(생산직 필수), **안전보건 법정관리**(검진·교육 미이수 경보) 강화, **고객 코드↔AEON 코드 매핑 도구**(종속용), **고객사 온보딩 마법사**.

### 2-4. 90일 로드맵 (권장)
| 기간 | 목표 |
|---|---|
| **0~30일** | MVP 스코프 확정 + Active 3파트 완성도 100% + 데모 데이터 정비 + E2E 녹화 |
| **30~60일** | 클라우드 배포 + 멀티테넌트 격리 + 보안/LLM 정책 + 급여밴드 구축 |
| **60~90일** | 고객 1곳 파일럿(원칙5) + 피드백 반영 + 종속 커넥터 1종(읽기연동) |

## 3. 1차 출시(MVP Launch) 프로세스 (매우 중요)

> "완벽"이 아니라 **"믿을 수 있는 좁은 범위"**를 출시한다.

### STEP 1 — 스코프 확정 (Freeze)
- 출시 파트 = **확보관리 + HR Synapse/Agent + 근태·노무·컴플라이언스**. (평가·교육·이직은 "로드맵"으로 표기, 화면은 두되 "미리보기" 배지)
- "이번 출시에 넣지 않는 것"을 문서로 못박음.

### STEP 2 — 완성도 다지기 (Harden)
- Active 파트의 빈 데이터·깨진 흐름 제거, 공통코드 미해석 정리, 급여밴드 값 확정.
- 빌드 통과 + pytest 통과 + 주요 화면 렌더 확인(증거 캡처).

### STEP 3 — 데모/시드 데이터
- 조직·직무·공고·지원자·면접·최종합격·직원·근태·노무사건·보상후보·컴플라이언스 기한 **현실 시드** 1세트.
- 대표 시나리오 1개 완주: **직무기준→공고→사람인→AI점수→면접→최종합격→입사전환→근태/계약/보상 영향**.

### STEP 4 — E2E 검증 (증거)
- Playwright로 위 시나리오 녹화 + 3모드(설계자/관리자/직원) 캡처 + 사람인 실행 영상.
- QA Gate Evidence Board L1~L3 채우기.

### STEP 5 — 보안·격리·감사
- Protected route·3모드·PII 마스킹·audit log 점검. 테넌트 분리. 비밀(.env) 관리.

### STEP 6 — 배포 (아래 4번 상세)
- 스테이징 → 프로덕션. 도메인·HTTPS·모니터링·백업.

### STEP 7 — 파일럿 & GA
- 고객사 1곳에 제한 오픈 → 피드백 2주 → 보수 → 정식 출시(GA).

### 출시 판정 체크리스트 (Go/No-Go)
- [ ] 대표 시나리오 1개 무결점 완주
- [ ] 빌드/테스트 통과 + 증거 캡처
- [ ] 3모드 경계 위반 없음
- [ ] 개인정보 마스킹·감사 작동
- [ ] 클라우드에서 외부인이 접속 가능
- [ ] 롤백 방법 존재

## 4. 웹에서 보려면 (배포 프로세스)

> 지금은 대표님 PC(로컬)에서만 돕니다. 인터넷에서 보려면 3덩어리를 각각 올려야 합니다.

### 4-1. 구성 (3덩어리 + 1워커)
| 덩어리 | 무엇 | 어디에 올리나(후보) |
|---|---|---|
| **프론트(화면)** | React 빌드 결과(정적 파일) | **Vercel** 또는 Netlify |
| **백엔드(두뇌)** | FastAPI 서버 | **Railway/Render/Fly.io** (Docker) |
| **DB(창고)** | Postgres | **Supabase**(이미 사용 중) |
| **사람인 워커** | Chrome로 이력서 수집 | ⚠️ 서버리스 불가 → **별도 VM(브라우저 있는 서버)** 또는 데스크톱 유지 |

### 4-2. 순서
1. **백엔드 배포**: Docker 이미지 → Railway/Render. 환경변수 `DATABASE_URL`(Supabase), `GEMINI_API_KEY` 설정 → `alembic upgrade head`로 DB 준비 → `uvicorn main:app` 기동. 공개 URL 확보(예: api.aeonhr.com).
2. **프론트 배포**: `npm run build` → Vercel 업로드. 환경변수 `VITE_API_URL`을 위 백엔드 URL로 지정.
3. **CORS/도메인**: 백엔드가 프론트 도메인 허용, HTTPS 자동(Vercel/Railway 기본).
4. **사람인 워커**: 브라우저가 필요하므로 별도 VM(예: 소형 클라우드 서버)에 배치하거나, 초기엔 데스크톱 실행 유지(비핵심화 금지 원칙 지키되 배포는 분리).
5. **운영**: 로그/모니터링, DB 백업, 비밀값은 코드에 넣지 말고 배포판 환경변수로.

### 4-3. 비개발자 요약
> **"화면은 Vercel, 두뇌는 Railway, 창고는 Supabase에 각각 올리고 서로 주소를 알려주면 인터넷에서 열립니다. 사람인 수집만 브라우저가 필요해 별도 서버가 듭니다."**

- 예상 난이도: 프론트·DB는 쉬움(반나절), 백엔드는 보통(1~2일), 사람인 워커가 가장 까다로움.
