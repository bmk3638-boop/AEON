## 1. 왜 시작했나 — 문제와 베팅

**문제**: 50~300인 제조·서비스업 HR 담당자는 채용·온보딩·평가·퇴직 전 사이클에서 반복 업무에 시간을 빼앗긴다. AI에 관심은 있지만 직접 구현할 시간·기술이 없다.

**AEON의 베팅**: 도구만 파는 게 아니라 **설계 → 구축 → 운영 이관** 전 과정을 책임진다. 그 결과물이 `hr-tools-web` = AEON HR OS.

**보통 회사 vs 우리**:
- 보통 회사: 채용/근태/평가/급여 기능을 각각 따로 만든다 → 데이터가 흩어지고 같은 걸 두 번 입력한다.
- AEON: "발령 한 번 = 조직·권한·급여·근태가 자동으로 따라 바뀌고, AI가 미리 위험을 짚고, 사람이 최종 승인한다."

## 2. 설계 헌법 — 제품의 DNA (세션14에서 제정)

기능을 쌓기 전에 **어떻게 만들 것인가**를 먼저 못 박았다. 이게 다른 HR SaaS와의 결정적 차이다.

**핵심 흐름** (모든 기능이 이 단계를 통과):

```
Information → Event → Synapse → Workflow → Decision → Record → Learning → Prediction → Autonomous HR Operation
```

**기능 만들 때 7가지 질문**: ①어떤 HR 정보? ②어떤 사건? ③어느 모듈로 전달? ④어떤 후속업무? ⑤사람은 무엇을 판단? ⑥판단 근거는 어떻게 기록? ⑦이 기록이 AI 학습재료가 되는가?

**6대 설계 원칙**:
1. 대시보드는 클릭하면 그 내용으로 바로 이동한다.
2. 칸반 카드는 클릭하면 상세·해야 할 액션이 보인다.
3. 한 모듈에 입력하면 연관 모듈로 자동 전파된다(단일 진실원천).
4. 관리자가 1분 안에 이해 못 하면 Before/After 설명을 붙이거나 설계를 바꾼다.
5. 내부 진화 후 외부 점검 — 완성=검증 대기. 고객 실증까지가 완료(컴플레인 제로).
6. 워크플로우는 반(半)다이나믹 — **AI가 다음 단계 제안 → 사람이 승인**. 자동적용 금지(컴플라이언스 영역).

**3모드 분리(절대 원칙)**: 설계자(AEON 내부) / 관리자(고객사 HR) / 사용자(고객사 직원). role 기반 modeScope로 경계, 경로만으로 판단 금지.

## 3. 구축 연대기 (Acts) — 처음부터 지금까지

### Act 1 — 토대: MVP와 채용 자동화 (5월 중순)

### Act 5 — 정체성 확립: 개발 헌법 + 3모드 (5/25)

## 4. 지금의 시스템 지도

**3모드**:
- **설계자모드(AEON HQ)** — platform_operator. QA Gate, Tenant Matrix, 고문단, Lab.
- **관리자모드(HR OS)** — 고객사 HR. 확보·조직·직무·교육·평가·보상·유지·이직 OS + 발령/증명서/라이프케어/마스킹.
- **사용자모드(My Page)** — 고객사 직원 셀프서비스.

**HR OS 모듈**(관리자): 확보관리(Acquisition)·조직관리(Organization)·직무관리(Job)·교육경력(Growth)·평가(Performance)·보상(Reward)·유지관리(Relations)·이직관리(Separation)·근태(Attendance)·노무(Labor)·컴플라이언스·HR Analytics.

**운영지능 인프라**: HR Synapse / Context Radar(맥락 레이더) / Agent Action 큐(반다이나믹 승인) / 알림함(DB+SSE) / 감사로그(Explainability).

**데이터**: PostgreSQL(Supabase) + SQLModel 44모델 + Alembic 22마이그레이션. AI는 Claude(Anthropic) 통일, 모든 산출물=초안(사람 승인 전제).
