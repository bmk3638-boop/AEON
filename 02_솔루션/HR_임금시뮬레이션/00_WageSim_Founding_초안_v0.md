# AEON 임금 시뮬레이션 — WageSim (Founding 초안 v0)

> 별명: **임금 플라이트 시뮬레이터**. 레버를 당기면 우리 회사 향후 인건비가 다시 계산된다.
> 이 문서가 BMAD의 입력(Product Brief)이다. → PRD → Architecture → Epics → Stories → QA로 전개.

---

## 0. 한 줄 정의

> **"우리 회사 임금·인력 구조에서 한 조건(통상임금·정년·최저임금 등)이 바뀌면 향후 N년 인건비가 얼마나 변하는가"를, 개인정보를 회사 밖으로 내보내지 않고, 30초 안에 금액으로 답하는 시나리오 엔진.**

핵심 한 줄: **"HR 전체 디지털 트윈"이 아니다. "단일 질문 엔진"의 집합이다.** (정밀함은 기능이 아니라 스코프에서 나온다.)

---

## 1. 왜 지금 — 모두의 창업 재도전 + 자격 로드맵

- **시장 타이밍**: 2024.12 대법원 통상임금 판결, 2033 정년연장 법정화, 매년 최저임금·주52h — 모든 중소·중견 대표가 "그래서 우리 회사 인건비 얼마 늘어?"를 묻는데 답해주는 도구가 없다.
- **피칭 킬러**: 화면에 즉시 **억대 리스크 금액**이 뜬다. 심사위원·CFO가 30초 만에 반응한다.
- **자격 로드맵(변경)**: 파트너 노무사 내장은 보류. **배명규 대표가 내년 공인노무사 취득 예정** → 본인이 자격자가 됨.
  - **그때까지 포지셔닝**: WageSim은 **"계산·What-if 시뮬레이터"**다. "노무 자문"이라 말하지 않는다. 수당의 통상임금 포함 여부 등 **법적 판단 입력값은 고객이 책임지고 입력**(또는 참고 기준표 제공)하며, 우리는 그 입력 위에서 **계산만** 한다. → 무자격 자문 리스크 회피.
  - **자격 취득 후(내년)**: "계산 도구 + 자격자 검증"으로 업그레이드. 분류 기준표에 대표 자격으로 서명 가능 → 단가·신뢰 점프.

## 2. 스코프 결정 (★ 가장 중요)

데모(콕핏)와 비판적 검토 문서 사이의 충돌을 이렇게 정리한다:

| | 콕핏 데모 | 비판적 검토 | 결론 |
|---|---|---|---|
| 그림 | 709명 10년 디지털 트윈 | "그 구축은 스케일 안 됨" | **콕핏=UI·피칭 무기로 유지** |
| 본질 | (화려함) | 단일 질문 엔진 + 플레이북 | **백엔드 모델은 비판적 검토대로** |
| 구축 | 1사 4~7주 핸드메이드 | 입력 템플릿으로 1~2주 | **고객이 채워오게 → 2주** |

→ **콕핏은 그대로, 그 뒤 엔진을 "질문 단위 + 분류 룰 + 로컬 전처리"로 정밀하게 짠다.**

## 3. ★ 시뮬레이션 로드맵 1~10 (이번 프로젝트의 척추)

> 순서 원칙: **(a) 피칭 임팩트 × (b) 시장 반복수요 × (c) 플랫폼 재사용도**.
> **#1을 만들면 플랫폼의 70%(로컬 전처리기·분포 모델·콕핏·수당 분류)가 완성된다. #2~#10은 "시나리오 모듈 +1"씩 얹는 것.**

### Tier A — 리스크 금액 (CFO 즉답 / 피칭 킬러) — 먼저

**1. 통상임금 재산정** ★ **v1 — 무조건 첫 번째**
- 질문: *정기상여가 통상임금에 포함되면 우리 회사 연 인건비 증가분은? 과거 3년 소급 부채는?*
- 출력: **연 +N억 / 소급 부채 N억** (즉시 화면)
- 왜 1번: 가장 토픽컬(대법원 판결) + 억대 숫자 즉답 + 이걸 만들면 플랫폼이 선다.

**2. 정년 연장 인건비**
- 질문: *60→63→65 단계 적용 시 향후 10년 인건비 곡선은?*
- 출력: 10년 누적 차액 / 연도별 곡선 (콕핏 이미 구현)
- 왜 2번: 2033 법정 이슈, 콕핏 자산 재사용, 모든 제조업 대표 관심.

**3. 퇴직금 충당부채**
- 질문: *정년연장·임금인상 시 DB형 퇴직급여 충당부채는 얼마나 늘어나나?*
- 출력: 충당부채 증가분 (재무제표 임팩트)
- 왜 3번: #2와 근속·정년 데이터 공유, CFO·회계 임팩트.

### Tier B — 정책 대응 (매년 반복 / 넓은 시장) — 다음

**4. 최저임금 인상 영향**
- 질문: *내년 최저임금 인상 시 영향 받는 인원·금액은? (통상임금 산입 연동)*
- 출력: 영향 인원 N명 / 추가 인건비 N원
- 왜 4번: 매년 반복 수요, 전 중소기업 대상으로 시장 가장 넓음.

**5. 주4일제 / 근로시간 단축**
- 질문: *주4일제(또는 주 40→36h) 도입 시 인건비·추가 채용 필요 인원은?*
- 출력: 인건비 변동 / 추가 채용 N명
- 왜 5번: 트렌디·정책 이슈, 피칭 화제성.

**6. 임금피크제**
- 질문: *56세부터 연 X% 감액 시 절감액 vs 고령자고용법 리스크는?*
- 출력: 절감액 / 리스크 플래그 (콕핏 레버 이미 있음)

### Tier C — 구조 전환 (대형 의사결정) — 그 다음

**7. 호봉제 → 직무급 전환**
- 질문: *호봉제를 직무급으로 바꾸면 인건비 변동과 승자/패자 분포는?*
- 출력: 총인건비 변동 / 직원별 증감 분포 히트맵
- 왜 7번: 큰 의사결정, 우리 HR OS의 직무관리·보상과 직결.

**8. 베이스업 / 임금협상 시나리오**
- 질문: *연 X% 인상 누적 효과 + 노사 협상안 A/B 비교는?*
- 출력: 시나리오 비교 차트 (협상 테이블용)

### Tier D — 인력 구조 (조직 미래) — 마지막

**9. 인력 고령화 + 정년 결원 충원**
- 질문: *10년 후 연령 분포는? 퇴직 1명을 신입으로 충원 시 인건비 절감은?*
- 출력: 연령 곡선 / 충원율별 비용 (콕핏 레버 있음)

**10. 이직률 / 적정 채용 시나리오**
- 질문: *이직률 X% 변화 시 채용비용·생산성 손실은? 적정 채용 규모는?*
- 출력: 이직 비용 / 권장 채용 N명
- 왜 10번: 우리 HR OS의 이직·확보 모듈과 결합 지점 (모듈 흡수 시 시너지).

> **요약**: #1 통상임금으로 플랫폼을 세우고 → Tier B로 시장을 넓히고 → Tier C/D로 큰 의사결정·미래를 덮는다.

## 4. 4계층 아키텍처

```
L1. 표현(Cockpit)      레버·차트·시나리오 A/B 비교·리포트 출력 + 입력 템플릿 폼
        ▲                         (데모 HTML로 이미 증명)
L2. 시나리오 엔진       10개 시뮬레이션 계산 로직 / 조합·비교 / 10년 곡선 산출
        ▲                         ("단일 질문 엔진"들의 집합)
L3. 도메인·룰          임금 모델(기본급·통상임금·수당·공제·퇴직금) + 수당 분류 룰셋
        ▲              + 법령/판례 파라미터(최저임금·정년·대법원) + 검증 룰
        ▲                         (← 여기에 RAG로 법령·판례 검색 연결 가능)
L4. 데이터·수집        로컬 전처리기(개인정보 익명화·분포 집계) + 분포 데이터 모델
                                   + 데모 데이터 생성기(709명 합성, 이미 있음)
```

- **PII 경계 = L4와 L1~L3 사이**: L4(로컬 전처리기)는 고객사 안에서 돌고, **개인 레코드 0건 / 직군×연령×호봉 분포 + 수당 합계만** 위로 올린다. → "개인정보는 귀사 방화벽 밖으로 안 나갑니다."
- 비판적 검토의 **2단 아키텍처(로컬/클라우드)**가 이 4계층에 매핑: **L4=로컬, L1~L3=클라우드.**
- L3에 RAG(법령·판례 지식검색)를 붙이면, 직전 대화의 RAG/BMAD 논의와도 연결된다(파라미터·근거 자동 갱신).

## 5. BMAD 연결 (이 문서 → 개발 산출물)

- 이 founding 초안 = **Product Brief** 역할.
- BMAD 전개: **PRD → Architecture(4계층) → Epics(=시뮬레이션 1~10) → Stories → QA**.
- Epic 매핑: **시뮬레이션 1개 = Epic 1개.** v1 = Epic 1(통상임금) → Story로 분해.
- 원칙6(반다이나믹) 적용: 시뮬레이션은 본질적으로 **What-if(자동 적용 없음)** → AI가 "정책 변경 제안" 시 사람 승인 게이트. (HR OS 흡수 시 RewardOS Synapse와 동일 패턴.)

## 6. 데이터 자산 — 보유 vs 필요

- ✅ **지금 충분(피칭·데모)**: 인사마스터 709명(익명), 급여 33컬럼 스키마, 콕핏·아카이브 데모 생성기, 통상임금 판례.
- 🔴 **구축에 필요(우선순위)**: 호봉테이블 원본 / 최근 3~5년 인상이력 / 수당별 지급조건(취업규칙) / 퇴직연금 DB·DC 비율.
- ⚠️ **철회**: 미승인 실데이터 데모 금지. 실데이터는 L4 로컬 전처리기 통과한 **분포만**.

## 7. v1 범위 (정밀하게 좁게)

> **통상임금 재산정 "한 질문"만 답하는 엔진 + 수당 분류표 + 로컬 전처리기 v0 + 콕핏 재사용.**

- 검증 기준: 1사 파일럿 구축을 **2주 안에** 완료 (비판적 검토의 "4~7주→2주" 증명).
- 산출물 = **구축 플레이북 3종**(입력 템플릿 / 수당 분류 기준표 / 검증 체크리스트). ← 이게 진짜 자산.

## 8. 과금 모델

- **구축 프로젝트(1회) + 연간 라이센스(시나리오 무제한·법령/판례 반영) + 이벤트 진단.**
- 과금 변수: **seat 금지 → 시뮬레이션 대상 인원수(데이터 규모).**
- 가격 앵커: 급여 SaaS 아님 → **노무법인 컨설팅 단가.**
