# HR Tools 작업 인수인계 2026-05-05

## 현재 판단

`hr-tools-web`는 React/Vite 프론트엔드와 FastAPI 백엔드가 분리된 HR AI 자동화 MVP다. 제품의 중심은 단순 HR 포털이 아니라 [[domains/채용자동화]]를 중심으로 한 사람인 채용 자동화, 지원자 관리, 평가/면접, AI 챗봇 데모 플로우다.

개발 서버 구조는 다음 기준으로 보는 것이 가장 안정적이다.

```text
Frontend: http://localhost:5173
Backend:  http://localhost:8000
```

현재 코드에는 포트 설정 불일치가 있다. `frontend/vite.config.js`는 `/api` 요청을 `http://localhost:8010`으로 프록시하도록 되어 있으나, 현재 백엔드 실행 문서와 `backend/start.bat`, Dockerfile 기준 포트는 `8000`이다. 이 상태에서는 프론트 화면이 떠도 로그인, 챗봇, 채용 API 호출이 실패할 수 있다.

## 확인된 구조

- 프론트엔드: React, Vite, Tailwind CSS
- 백엔드: FastAPI, Uvicorn
- AI 챗봇: Gemini API
- 주요 화면: Home, Login, Signup, Recruiting, Dashboard, ApplicantList, EvaluationForm, InterviewSchedule, LaborQA, OCRDemo, PerformanceMgmt, BioData
- 주요 API: `/health`, `/api/chat`, `/api/auth/*`, `/api/recruiting/*`, `/api/applicants/*`, `/api/employees/*`

## 확인된 리스크

- `frontend/Dockerfile`과 `frontend/nginx.conf`가 현재 체크아웃에 없다. `docker compose up -d --build`는 이 상태에서 실패할 가능성이 높다.
- 루트 `.env`가 없다. `GEMINI_API_KEY`가 없으면 챗봇은 실제 Gemini 응답을 반환하지 못한다.
- `ChatbotPanel.jsx` 등 일부 프론트 파일에서 한글 인코딩 깨짐과 JSX 문법 깨짐이 보인다.
- `OCRDemo.jsx`는 `/api/ocr/parse`를 호출하지만 백엔드 라우터 목록에서는 OCR API가 확인되지 않는다.
- API 호출 방식이 혼재되어 있다. 일부는 `VITE_API_URL`을 쓰고, 일부는 상대 경로와 Vite proxy에 의존한다.

## 다음 작업 우선순위

1. `5173 -> 8000` API 연결 기준을 확정한다.
2. `vite.config.js`, `.env.local`, 프론트 API base 사용 방식을 정리한다.
3. 깨진 JSX와 사용자 노출 한글 문자열을 복구한다.
4. 백엔드 라우트와 프론트 호출 목록을 대조해 누락 API를 정리한다.
5. 핵심 데모 플로우를 QA한다: 로그인 -> 사람인 연결 -> 채용 자동화 상태 -> 지원자 목록 -> 평가/면접 -> 챗봇.
6. Docker 실행을 복구한다: `frontend/Dockerfile`, `frontend/nginx.conf`, nginx `/api` 프록시.

## 작업 원칙

- AEON vault는 제품 방향과 장기 기억의 기준 문서다.
- `hr-tools-web` 저장소는 실행 가능한 구현의 기준이다.
- `saramin-automation` 원본 폴더는 직접 수정하지 않는다. 복사본 또는 `backend/modules/recruiting`만 작업한다.
- 사용자 화면에는 백엔드 로그, 콘솔, 디버그 메시지를 노출하지 않는다.
- `.env`와 API 키는 커밋하지 않는다.

## Codex와 Claude Code 역할분담

- Codex: 코드 구조 분석, 실행 환경 정리, 버그 수정, API 연결, Docker 복구, QA.
- Claude Code: AEON vault 기반 기획 맥락, UX 문구, 문서화, 장기 방향성 정리, 제품 리뷰.
- 공통 기준: `AGENTS.md`, `CLAUDE.md`, AEON vault의 사업/솔루션 문서를 우선 참고한다.

## md 파일을 나열한 이유

초기 응답에서 관련 `.md` 파일을 나열한 이유는 작업의 기준 문서를 식별하기 위해서다. 이 프로젝트는 코드만 보면 "React + FastAPI 앱"으로 보이지만, AEON vault까지 보면 "HR AI 자동화 컨설팅에서 SaaS MVP로 전환하는 제품"이라는 의도가 드러난다. 따라서 `CLAUDE.md`, `OBSIDIAN_HUB.md`, `RECIPE.md`, `wiki/entities/hr-tools-web.md`, `wiki/domains/채용자동화.md` 같은 문서는 단순 문서가 아니라 에이전트들이 같은 방향으로 일하기 위한 제품 메모리다.

앞으로는 문서 목록을 길게 던지는 방식보다, 먼저 결론과 판단을 말하고 필요한 문서 링크만 근거로 붙이는 방식이 더 적합하다.
